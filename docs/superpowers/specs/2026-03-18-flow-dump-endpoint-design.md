# Flow Dump Endpoint Design

**Date:** 2026-03-18
**Issue:** #16
**Status:** Approved

## Overview

Add `GET /v1/flows/dump` — a single endpoint that returns full flow data (steps, edges, stats, metadata) with filtering, pagination, and format options. Serves analyst export, integration pipelines, and bulk analysis use cases.

## Endpoint

```
GET /v1/flows/dump
```

**Route ordering:** This route MUST be registered before `GET /v1/flows/{flow_id}` in the router to avoid FastAPI's greedy path parameter matching capturing `/dump` as a flow_id.

### Query Parameters

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `report_id` | string | — | Filter by source report STIX ID |
| `actor` | string | — | Filter by threat actor name (fuzzy match) |
| `actor_id` | string | — | Filter by intrusion set STIX ID |
| `flow_type` | string | — | `deterministic` or `llm_synthesized` (mapped to stored values internally) |
| `technique` | string | — | Filter flows containing technique (e.g. `T1566.001`) |
| `ingested_after` | date | — | Filter reports ingested after this date (ISO 8601) |
| `ingested_before` | date | — | Filter reports ingested before this date (ISO 8601) |
| `format` | string | `json` | Response format: `json` or `stix` |
| `limit` | int | 50 | Results per page (max 200) |
| `offset` | int | 0 | Number of results to skip |

#### `flow_type` Values

The OpenSearch `extraction.flows[].flow_type` field is normalized before storage by `job_processor.py`, `extraction_pipeline.py`, and `reports.py` to one of:

| API/Stored Value | Description |
|------------------|-------------|
| `deterministic` | Built from tactic ordering + narrative position |
| `llm_synthesized` | Generated by LLM flow synthesis |
| `co-occurrence` | Co-occurrence-based flow (no temporal ordering) |

Note: Raw flow builders may use different internal values (`deterministic_full`, `sequential`) but these are normalized to the values above before OpenSearch storage.

### Response — STIX Format

When `?format=stix`, each flow is exported via `AttackFlowExporter` into Attack Flow 2.0 STIX bundles. Only flows persisted to Neo4j can be exported as STIX (requires graph structure). Flows without Neo4j backing are omitted from STIX output. The `total` field reflects all matching flows (including non-exportable ones), while `exported_count` indicates how many STIX bundles are included.

```json
{
  "flows": [
    {
      "flow_id": "uuid",
      "source_id": "report--xxx",
      "stix_bundle": {
        "type": "bundle",
        "id": "bundle--uuid",
        "objects": [...]
      }
    }
  ],
  "total": 124,
  "exported_count": 80,
  "limit": 50,
  "offset": 0
}
```

## Data Source Strategy

### Primary: OpenSearch

Query `bandjacks_reports` index where flows are stored inline in `extraction.flows[]`. This avoids N+1 Neo4j queries and gives us complete step/edge data in a single query.

### Filter Implementation

All filters except `report_id`, `ingested_after`/`ingested_before`, and `actor`/`actor_id` are **application-side post-query filters** applied after fetching report documents. The `extraction.flows` array is dynamically mapped in OpenSearch (not explicitly indexed), so field-level queries against nested flow properties are not reliable.

| Filter | Strategy |
|--------|----------|
| `report_id` | OpenSearch `term` filter on `report_id` field |
| `ingested_after` / `ingested_before` | OpenSearch `range` filter on `ingested_at` field |
| `actor` / `actor_id` | Neo4j lookup: `MATCH (r:Report)-[:IDENTIFIED_ACTOR]->(g:IntrusionSet)` to get report IDs, then OpenSearch `terms` filter on those IDs |
| `flow_type` | Application-side: filter `extraction.flows[]` by `flow_type` after fetch |
| `technique` | Application-side: scan `extraction.flows[].steps[].attack_pattern_ref` after fetch |

**Performance implication:** Without filters that reduce the OpenSearch result set (report_id, date range, actor), the endpoint fetches all reports and filters in memory. For large deployments, at least one server-side filter should be encouraged in documentation.

### Pagination Strategy

Since flows are unnested from reports in application code, pagination operates on the **flat list of flows** after post-query filtering:

1. Fetch reports from OpenSearch (with server-side filters applied, fetch up to `10 * limit` reports to ensure enough flows)
2. Unnest `extraction.flows[]` from each report into flat list
3. Apply post-query filters (flow_type, technique)
4. Count total matching flows
5. Apply offset/limit slice
6. Return paginated result

The `total` field is the **count of matching flows after all filters**, computed by scanning the full filtered set. For very large result sets (1000+ reports), this may require multiple OpenSearch scroll queries. The implementation should cap the scan at 10,000 reports and return `"total_truncated": true` if exceeded.

## Architecture

### Files Modified

1. **`bandjacks/services/api/routes/flows.py`** — New `GET /v1/flows/dump` route handler
   - MUST be registered before `GET /{flow_id}` to avoid path conflict
   - Validates and parses query parameters
   - Maps `flow_type` API values to stored values
   - Resolves actor filters via Neo4j if needed
   - Calls `dump_flows()` on the report store
   - Applies post-query filters (flow_type, technique)
   - Handles STIX format conversion via AttackFlowExporter
   - Returns paginated response

2. **`bandjacks/store/opensearch_report_store.py`** — New `dump_flows()` method
   - Builds OpenSearch query with server-side filters (report_id, date range)
   - Fetches report documents including `extraction.flows` and `name`
   - Unnests flows from report documents into flat list, attaching `report_name` and `ingested_at`
   - Returns raw unnested flows (post-query filtering done by caller)

### Flow of Data

```
Request → Route Handler
  ├── actor/actor_id filter? → Neo4j lookup → report_id list
  ├── Build OpenSearch query (report_id, date range)
  ├── dump_flows() → fetch reports, unnest flows
  ├── Apply post-query filters (flow_type, technique)
  ├── Count total, apply offset/limit
  ├── format=stix? → AttackFlowExporter per flow (Neo4j lookup)
  └── Return response with total, limit, offset
```

## Error Handling

- Invalid `format` value → 400 with message
- Invalid `flow_type` value → 400 with allowed values listed
- `limit` > 200 → clamp to 200
- `actor`/`actor_id` not found in Neo4j → return empty results (not 404)
- STIX export failure for individual flow → include `"stix_bundle": null` with `"error"` field, don't fail the whole response
- OpenSearch index missing → return empty results with warning
- Result set exceeds 10,000 reports → return results with `"total_truncated": true`

## Testing

- Unit test for `dump_flows()` query building and flow unnesting
- Integration test: ingest report, verify flow appears in dump
- Filter tests: each filter individually and in combination
- `flow_type` mapping test: verify `deterministic` matches `deterministic_full`
- Pagination test: verify offset/limit/total consistency across report boundaries
- STIX format test: verify valid STIX bundle output for Neo4j-backed flows
- Edge case: report with 0 flows, report with 2 flows, empty database
