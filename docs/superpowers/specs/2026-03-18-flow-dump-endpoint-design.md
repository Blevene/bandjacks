# Flow Dump Endpoint Design

**Date:** 2026-03-18
**Issue:** #16
**Status:** Approved

## Overview

Add `GET /v1/flows/dump` — a single endpoint that returns full flow data (steps, edges, stats, metadata) with filtering, pagination, and format options. Serves analyst export, integration pipelines, and bulk analysis use cases.

## Endpoint

```
GET /v1/flows/dump
```

**Route ordering:** This route MUST be registered before `GET /v1/flows/{flow_id}` in the router to avoid FastAPI's greedy path parameter matching capturing `/dump` as a flow_id.

### Query Parameters

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `report_id` | string | — | Filter by source report STIX ID |
| `actor` | string | — | Filter by threat actor name (case-insensitive substring match) |
| `actor_id` | string | — | Filter by intrusion set STIX ID (exact match) |
| `campaign` | string | — | Filter by campaign name (case-insensitive substring match) |
| `campaign_id` | string | — | Filter by campaign STIX ID (exact match) |
| `flow_type` | string | — | `deterministic`, `llm_synthesized`, or `co-occurrence` |
| `technique` | string | — | Filter flows containing technique by ATT&CK ID (e.g. `T1566.001`) |
| `ingested_after` | date | — | Filter reports ingested after this date (ISO 8601, UTC assumed) |
| `ingested_before` | date | — | Filter reports ingested before this date (ISO 8601, UTC assumed) |
| `format` | string | `json` | Response format: `json` or `stix` |
| `limit` | int | 50 | Results per page (max 200) |
| `offset` | int | 0 | Number of results to skip |

#### `flow_type` Values

The OpenSearch `extraction.flows[].flow_type` field is normalized before storage by `job_processor.py`, `extraction_pipeline.py`, and `reports.py` to one of:

| API/Stored Value | Description |
|------------------|-------------|
| `deterministic` | Built from tactic ordering + narrative position |
| `llm_synthesized` | Generated by LLM flow synthesis |
| `co-occurrence` | Co-occurrence-based flow (no temporal ordering) |

Note: Raw flow builders may use different internal values (`deterministic_full`, `sequential`) but these are normalized to the values above before OpenSearch storage. As a defensive measure, the `flow_type` filter should also match `deterministic_full` when `deterministic` is requested, in case any code path bypasses normalization.

#### `technique` Filter Matching

The `technique` parameter accepts an ATT&CK ID (e.g. `T1566.001`). It matches against `steps[].attack_pattern_ref` by checking if the ref ends with the provided ID (e.g. `attack-pattern--T1566.001` matches `T1566.001`). This is a suffix match on the STIX ID.


**Field notes:**
- `report_name`: sourced from the report document's `name` field
- `confidence`: flow-level confidence (0.0–1.0 scale, always present, defaults to `0.5`)
- `steps[].confidence`: step-level confidence (0–100 scale, from extraction claims)
- `ingested_at`: the report's `ingested_at` timestamp (when the report was processed)
- `steps[].tactic_rank`: kill chain position integer from `flow_deterministic.py` (e.g. initial-access=3, execution=4)
- `steps[].narrative_position`: earliest line reference from source text; `999999` if no line refs
- `edges[].tier`: edge confidence tier (`high`/`medium`/`low`) computed by `DeterministicFlowBuilder._build_edges()` based on tactic progression and narrative position
- `stats.tier_distribution`: count of edges per tier
- Steps and edges are returned as stored in `extraction.flows[]` — these are the raw action/edge dicts from `DeterministicFlowBuilder` or `FlowSynthesizer`

### Response — STIX Format

When `?format=stix`, each flow is exported via `AttackFlowExporter` into Attack Flow 2.0 STIX bundles. Only flows persisted to Neo4j can be exported as STIX (requires graph structure). A flow is considered Neo4j-backed if it has a `flow_id` that exists as an `AttackEpisode.flow_id` node. Flows without Neo4j backing are omitted from STIX output.

The `total` field reflects all matching flows (including non-exportable ones). The `exported_count` field indicates how many STIX bundles are actually included in the response.

```json
{
  "flows": [
    {
      "flow_id": "uuid",
      "source_id": "report--xxx",
      "stix_bundle": {
        "type": "bundle",
        "id": "bundle--uuid",
        "objects": [...]
      }
    }
  ],
  "total": 124,
  "exported_count": 80,
  "limit": 50,
  "offset": 0
}
```

## Data Source Strategy

### Primary: OpenSearch

Query `bandjacks_reports` index where flows are stored inline in `extraction.flows[]`. This avoids N+1 Neo4j queries and gives us complete step/edge data in a single query.

**Source projection:** Fetch only needed fields to avoid loading heavy data:
```python
"_source": {
    "includes": ["report_id", "name", "ingested_at", "extraction.flows"],
    "excludes": ["raw_text", "text_chunks", "chunk_embeddings", "extraction.bundle", "extraction.claims"]
}
```

### Filter Implementation

All filters except `report_id`, `ingested_after`/`ingested_before`, `actor`/`actor_id`, and `campaign`/`campaign_id` are **application-side post-query filters** applied after fetching report documents. The `extraction.flows` array is dynamically mapped in OpenSearch (not explicitly indexed), so field-level queries against nested flow properties are not reliable.

| Filter | Strategy |
|--------|----------|
| `report_id` | OpenSearch `term` filter on `report_id` field |
| `ingested_after` / `ingested_before` | OpenSearch `range` filter on `ingested_at` field |
| `actor` / `actor_id` | Neo4j lookup to get report IDs, then OpenSearch `terms` filter |
| `campaign` / `campaign_id` | Neo4j lookup: `MATCH (r:Report)-[:DESCRIBES_CAMPAIGN]->(c:Campaign)` to get report IDs, then OpenSearch `terms` filter |
| `flow_type` | Application-side: filter `extraction.flows[]` by `flow_type` after fetch |
| `technique` | Application-side: suffix-match `steps[].attack_pattern_ref` against provided ATT&CK ID |

**Actor lookup query:**
```cypher
MATCH (r:Report)-[:IDENTIFIED_ACTOR]->(g:IntrusionSet)
WHERE g.stix_id = $actor_id OR toLower(g.name) CONTAINS toLower($actor)
RETURN r.stix_id AS report_id
```

**Campaign lookup query:**
```cypher
MATCH (r:Report)-[:DESCRIBES_CAMPAIGN]->(c:Campaign)
WHERE c.stix_id = $campaign_id OR toLower(c.name) CONTAINS toLower($campaign)
RETURN r.stix_id AS report_id
```

**Performance implication:** Without filters that reduce the OpenSearch result set (report_id, date range, actor, campaign), the endpoint fetches all reports and filters in memory. For large deployments, at least one server-side filter should be encouraged in documentation.

### Pagination Strategy

Since flows are unnested from reports in application code, pagination operates on the **flat list of flows** after post-query filtering.

**Sort order:** Reports are fetched from OpenSearch sorted by `ingested_at DESC, report_id ASC` to ensure stable pagination across requests.

**Algorithm:**
1. Fetch all matching reports from OpenSearch (with server-side filters, capped at 10,000 reports using `search_after` for efficient scrolling)
2. Unnest `extraction.flows[]` from each report into flat list, attaching `report_name` and `ingested_at`
3. Apply post-query filters (flow_type, technique)
4. Count total matching flows
5. Apply offset/limit slice
6. Return paginated result

The `total` field is the **count of matching flows after all filters**. If the report scan hits the 10,000-report cap, the response includes `"total_truncated": true` to signal that more flows may exist.

## Architecture

### Files Modified

1. **`bandjacks/services/api/routes/flows.py`** — New `GET /v1/flows/dump` route handler
   - MUST be registered before `GET /{flow_id}` to avoid path conflict
   - Validates and parses query parameters
   - Maps `flow_type` API values to stored values (including defensive `deterministic_full` matching)
   - Resolves actor/campaign filters via Neo4j if needed
   - Calls `dump_flows()` on the report store
   - Applies post-query filters (flow_type, technique)
   - Handles STIX format conversion via AttackFlowExporter
   - Returns paginated response
   - Obtains OpenSearch client via `Depends(get_opensearch_client)` and instantiates `OpenSearchReportStore` in the handler (matching existing pattern in `routes/reports.py`)

2. **`bandjacks/store/opensearch_report_store.py`** — New `dump_flows()` method
   ```python
   def dump_flows(
       self,
       report_ids: Optional[List[str]] = None,
       ingested_after: Optional[str] = None,
       ingested_before: Optional[str] = None,
       max_reports: int = 10000,
   ) -> Tuple[List[Dict[str, Any]], bool]:
       """
       Fetch flows from report documents, unnested into a flat list.

       Args:
           report_ids: Optional list of report IDs to filter by
           ingested_after: ISO 8601 date string for range filter
           ingested_before: ISO 8601 date string for range filter
           max_reports: Maximum reports to scan (default 10,000)

       Returns:
           Tuple of (list of flow dicts with report_name/ingested_at attached, truncated bool)
       """
   ```

### Flow of Data

```
Request → Route Handler
  ├── actor/campaign filter? → Neo4j lookup → report_id list
  ├── Build server-side filters (report_id, date range)
  ├── dump_flows() → fetch reports sorted by ingested_at DESC, unnest flows
  ├── Apply post-query filters (flow_type, technique)
  ├── Count total, apply offset/limit
  ├── format=stix? → batch check Neo4j for flow_ids, export via AttackFlowExporter
  └── Return response with total, limit, offset, filters_applied
```

## Error Handling

- Invalid `format` value → 400 with message
- Invalid `flow_type` value → 400 with allowed values listed (`deterministic`, `llm_synthesized`, `co-occurrence`)
- Invalid `ingested_after`/`ingested_before` date → 400 with message
- `limit` > 200 → clamp to 200
- `actor`/`actor_id` not found in Neo4j → return empty results (not 404)
- `campaign`/`campaign_id` not found in Neo4j → return empty results (not 404)
- STIX export failure for individual flow → include `"stix_bundle": null` with `"error"` field, don't fail the whole response
- OpenSearch index missing → return empty results with warning
- Result set exceeds 10,000 reports → return results with `"total_truncated": true`

## Testing

- Unit test for `dump_flows()` query building and flow unnesting
- Integration test: ingest report, verify flow appears in dump
- Filter tests: each filter individually and in combination
- `flow_type` filter: verify `deterministic` matches both `deterministic` and `deterministic_full` stored values
- `technique` filter: verify suffix matching (`T1566.001` matches `attack-pattern--T1566.001`)
- Campaign filter: verify campaign name and campaign_id both resolve correctly
- Pagination test: verify offset/limit/total consistency with stable sort order
- STIX format test: verify valid STIX bundle output for Neo4j-backed flows; verify `exported_count` < `total` when some flows lack Neo4j backing
- Edge case: report with 0 flows, report with 2 flows, empty database
